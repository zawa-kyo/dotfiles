# 本質的なコードレビュー

## 言語バージョン

このファイルと `SKILL.md` は同じ skill を説明します。どちらかを変更する場合は、同じ変更の中でもう一方も更新し、英語版と日本語版の内容をそろえます。その際、それぞれの言語として自然な表現を保ちます。

## 概要

まず通常のコードレビューの観点で、正しさ、リグレッション、エッジケース、セキュリティ、パフォーマンス、テストカバレッジを確認します。そのうえで、明示的に単純さの観点を加えます。つまり、そのコードがローカルな設計の流儀に沿って根本問題を解いているのか、それとも特殊ケース、重複ロジック、広すぎるガード、不要な抽象化で症状だけを塞いでいるのかを見ます。

## ワークフロー

1. ユーザーの依頼からレビュー対象を特定する。対象が指定されていない場合は、ユーザーが staged-only やより狭い対象を指定していない限り、現在の Git 差分をステージング済みと未ステージングに分けて確認する。
2. 既存の設計、不変条件、公開契約、ローカルスタイルを理解するために、十分な周辺コードを読む。文脈なしに単純さを判断しない。
3. ユーザーが明示的に修正を依頼していない限り、レビュー中にワークツリーや index を変更しない。特に、`git add`、`git restore --staged`、フォーマット、その他レビュー対象の差分を stage、unstage、書き換え、正規化するコマンドを実行しない。
4. まず具体的な正しさのリスクをレビューする。バグ、リグレッション、欠けているエッジケース、壊れた API 契約、データ損失、セキュリティ問題、パフォーマンス問題、不足または弱いテストを確認する。
5. 次に、本質的な設計をレビューする。
   - 変更は根本原因に対処しているか、それとも観測された失敗を抑え込んでいるだけか。
   - 既存の不変条件を保っているか、それとも散らばった例外処理を増やしているか。
   - 偶発的な複雑さを取り除いているか、それともテストを通すために flag、分岐、retry、catch、sleep、cast、global、重複ロジックを増やしているか。
   - その抽象化は、実際の再利用や複雑さの低減によって正当化されているか。
   - 呼び出し側で挙動が明確で、次のメンテナーが理解しやすいか。
6. 具体的な行と失敗モードを示せる指摘を優先する。保守性に影響するか、本当のリスクを隠している場合を除き、スタイル上の細かな指摘は避ける。
7. 実装が単純で堅実な場合は、そう明確に伝える。レビュー形式を満たすために指摘を作り出さない。

## 回避策の匂い

以下は自動的な欠陥ではなく、より深く確認するための手がかりとして扱います。

- 広すぎる `try/catch`、`return null`、デフォルト値、または guard によって、なぜ続行してよいのかを説明しないまま不正な状態を隠している。
- 近くのコードに既にある一般ルールを使わず、失敗した入力だけをハードコードで特別扱いしている。
- 変換や検証のロジックを、その責務を持つドメイン境界で再利用せずに重複させている。
- ある失敗経路を迂回するためだけに、新しい設定、flag、任意パラメータを追加している。
- 実証された並行性モデルやライフサイクルモデルなしに、タイミング依存の修正、retry、sleep、cache clear、順序依存を追加している。
- type assertion、未検査の cast、緩めた schema によって、契約を弱めながらコンパイラや validator の警告を止めている。
- テスト変更が assertion を減らす、挙動を skip する、または意図した挙動を証明せずに新しい出力を正解として固定している。

## 出力

標準的なコードレビューの形を使います。

- 重大度順に findings から始める。
- 各 finding には、ファイルと行の参照、重大度、具体的なリスク、最小限の方向性として正しい修正方針を含める。
- 読みやすくなる場合に限り、「correctness findings」と「simplicity/design findings」を分ける。
- findings の後に、未解決の質問や前提を書く。
- レビューの信頼度に影響する場合は、実行しなかった検証を明記する。
- 簡潔な summary は findings の後にだけ置く。
- ユーザーが明示的に別言語を求めていない限り、レビューはユーザーの依頼言語で書く。

問題が見つからない場合は、blocking issue は見つからなかったと明確に伝え、残るテストや文脈上の不足があれば触れます。

ユーザーが明示的に修正を依頼していない限り、コードを書き換えません。レビュー出力は、ユーザーが何をなぜ変えるべきか判断できる助けにします。

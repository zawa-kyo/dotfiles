# グローバルエージェントガイド

English version: [AGENTS.md](AGENTS.md)

## 適用範囲

- このファイルは、コーディングエージェント向けのグローバルなユーザー単位の指示です。
- プロジェクト単位の instruction file やユーザーの明示的な依頼がより具体的な場合は、そちらを優先します。
- リポジトリ固有の構成、コマンド、運用方針は、そのリポジトリ自身の instruction file や docs に置きます。
- instruction は作業上のガイドとして扱い、コード読解や動作確認の代わりにはしません。

## コミュニケーション

- 成果物に明示的な言語要件がある場合を除き、ユーザーの依頼言語で応答します。
- 途中報告と最終回答は簡潔にし、現在のタスクに必要な内容へ絞ります。
- 実装やレビュー結果に影響する前提がある場合は明示します。
- 情報不足で作業が止まる場合、または推測で進めるのが危険な場合だけ質問します。

## 作業スタイル

- 重要な変更を入れる前に、周辺コードとプロジェクト文書を確認します。
- 新しい流儀を持ち込むより、プロジェクト既存のパターン、ツール、補助 API を優先します。
- 変更はユーザーの依頼範囲に絞り、無関係なリファクタリングを避けます。
- instruction file に長い手順を複製せず、正となる参照先へリンクします。
- コメントは、意図が読み取りにくい処理を補足する場合だけ追加します。

## 編集

- dirty worktree を尊重し、自分が加えていない変更を明示的な依頼なしに戻したり上書きしたりしません。
- ユーザーが明確に求めていない破壊的なコマンドは避けます。
- フォーマッタ対象ファイルを編集したら、リポジトリのフォーマッタとスタイル規約に従います。
- 生成物、マシン固有値、秘密情報、ローカルパス、認証情報をコミット対象に含めません。

## 検証

- 変更に対して有効な最小限の確認を行い、リポジトリに検証方針がある場合はそれに従います。
- テストやチェックを実行しなかった場合は、その理由と残る制約を伝えます。
- ドキュメントのみの変更では、無関係なテスト一式よりも format やリンク・参照の確認を優先します。

## Git

- ユーザーが依頼しない限り、stage、unstage、commit、push は行いません。
- コミットメッセージを提案するときは、まず現在の Git 状態を確認します。
- ユーザーが別形式を求めない限り、簡潔な英語の Conventional Commit を優先します。

## レビュー

- コードレビューでは、具体的な指摘を重要度順に先に出します。
- 正しさ、回帰、テスト不足、データ損失、セキュリティ、保守性のリスクを優先します。
- blocking な問題がなければその旨を明確にし、残る検証上の不足があれば伝えます。

## instruction の管理

- このグローバルファイルは、毎回読み込まれてもプロジェクト文脈を圧迫しない長さに保ちます。
- タスク固有またはツール固有のワークフローは、skill、command、プロジェクト単位の docs に分けます。
- このファイルは、ほとんどのリポジトリに適用したい指示だけを更新対象にします。
